**src/utils/parseRoadmap.ts** — Create new file. Pure function: ROADMAP.md string -> PhaseData[].

Parse two sections from the ROADMAP.md content:

1. **Phase bullets** from `## Phases` section. Pattern: `- [x] **Phase N: Name** - description` or `- [ ] **Phase N: Name** - description`. Use regex: `/^- \[([ x])\] \*\*Phase (\d+(?:\.\d+)?): ([^*]+)\*\*/gm`. The `[x]` means complete, `[ ]` means not complete.

2. **Progress table** rows. Pattern: `| N. Name | X/Y | Status | ... |`. Use regex: `/^\|\s*(\d+(?:\.\d+)?)\.\s*[^|]+\|\s*(\d+)\/(\d+)\s*\|\s*([^|]+)\|/gm`. Extract plansComplete and plansTotal.

Status derivation:
- `bullet.checked === true` -> 'complete'
- `bullet.checked === false && plansComplete > 0` -> 'in-progress'
- `bullet.checked === false && plansComplete === 0` -> 'not-started'

Initialize `dirName: null` and `files: []` — these are filled in by useGsd after filesystem scan.

CRITICAL: Wrap entire function body in try/catch. Return `[]` on any parse failure. Never throw. This is defensive parsing — ROADMAP.md format may vary across user projects.

Export both `parseRoadmap` function and `PhaseData` type (re-export from types.ts if preferred, but the canonical definition lives in types.ts).

New function — `loadPlanning()`:
1. Set `planningLoading = true`.
2. Read ROADMAP.md via `shellRef.current.exec('bash', ['-c', 'cat "${project!.path}/.planning/ROADMAP.md" 2>/dev/null'])`.
3. Parse with `parseRoadmap(result.stdout)` to get `PhaseData[]`.
4. List phase directories via `shellRef.current.exec('bash', ['-c', 'ls -1 "${project!.path}/.planning/phases" 2>/dev/null || echo ""'])`.
5. Match directories to phases: for each directory matching `/^(\d+)-/`, find the phase where `Math.floor(phase.number) === parseInt(dirMatch[1], 10)` and set `phase.dirName = dir`.
6. For each phase with a dirName, list its files via `shellRef.current.exec('bash', ['-c', 'ls -1 "${project!.path}/.planning/phases/${phase.dirName}" 2>/dev/null || echo ""'])`. Filter to `.md` files. Set `phase.files = filteredFiles`.
7. Set `planningData = phases`, `planningLoading = false`.
8. Wrap in try/catch — on error, set planningData to [], planningLoading to false.

CRITICAL: Call `loadPlanning()` at the end of `detect()` when phase is set to `has-planning`. This keeps a single async chain. The call goes AFTER `setPhase('has-planning')` but BEFORE the `finally` block sets loading to false. Actually, better: call loadPlanning() right after `setPhase('has-planning')` inside the try block. loadPlanning manages its own planningLoading state independently of the main loading state.

New function — `readFile(relativePath: string)`:
1. Increment `fileReadIdRef.current` and store as `requestId`.
2. Set `fileLoading = true`, `activeFile = null`, `fileContent = null`.
3. Read file via `shellRef.current.exec('bash', ['-c', 'cat "${project!.path}/${relativePath}" 2>/dev/null'], { timeout: 10000 })`.
4. Guard: if `requestId !== fileReadIdRef.current`, return (stale request).
5. If exit_code !== 0 or stdout is empty: show error toast, keep file states null.
6. If successful: `setActiveFile(relativePath)`, `setFileContent(result.stdout)`.
7. In finally: only set `fileLoading = false` if `requestId === fileReadIdRef.current`.
8. Make this a useCallback with [project] dependency (same pattern as detect).

New function — `clearFileView()`:
1. `setActiveFile(null)`, `setFileContent(null)`.
2. Make this a useCallback with no dependencies.

New in return object: add `planningData`, `planningLoading`, `activeFile`, `fileContent`, `fileLoading`, `readFile`, `clearFileView`, `showToast: actionsRef.current.showToast`.
